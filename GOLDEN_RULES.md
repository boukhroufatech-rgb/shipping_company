## 🌳 القاعدة الأولى — نظام الشجرة (The Tree System)
**المبدأ:** البرنامج يعمل بنظام هرمي (Hierarchical System).
- **الجذر (Root):** المكونات الأساسية الموحدة (`EnhancedTableView`, `SmartFormDialog`, إلخ).
- **الفروع (Branches):** كل وحدة (Module) تستخدم الجذر وترث خصائصه.
- **القاعدة:** إذا تغير الجذر → يتغير كل شيء تلقائياً. هذا هو الهدف.

```
تغيير في EnhancedTableView
        ↓
يؤثر تلقائياً على:
  - Treasury  - Clients  - Logistics
  - Partners  - Licenses - ExternalDebt
```

**⛔ ممنوع:** تعديل سلوك الأزرار/الألوان/النصوص مباشرة في كل Module بشكل يدوي.
**✅ الصواب:** تعديل المكون الأساسي (الجذر) لينعكس على الكل.

## ⚙️ القاعدة الثانية — الآلي vs المخصص (Automatic vs Custom)
**هناك نوعان فقط من الكود المقبول:**

### 1. الآلي (Automatic)
- يعمل عبر الشجرة (Tree).
- لا يحتاج توثيقاً خاصاً لأنه يتبع القاعدة العامة.
- مثال: `EnhancedTableView` في كل الوحدات.

### 2. المخصص (Custom)
- مشتق من الآلي لكن يحتاج تعديلاً خاصاً بسبب متطلب معين.
- **يجب توثيقه بعلامة التعليق التالية:**

```python
# [CUSTOM] السبب: ...
# [WHY]: ...
# [DATE]: YYYY-MM-DD
```

**مثال:**
```python
# [CUSTOM] btn_smart_activate: زر الاقتراح الذكي في مكتبة العملات
# [WHY]: هذا الزر يعمل بمنطق "Smart Discovery" ولا يشبه أي زر قياسي في النظام.
#        يظهر فقط عند الكتابة في البحث ويقترح عملة من الكتالوج العالمي.
# [DATE]: 2026-03-29
self.btn_smart_activate = QPushButton("Activer maintenant")
```

## 🚫 القاعدة الثالثة — ممنوع اليدوي بدون توثيق (No Undocumented Manual Code)
- أي `QPushButton` أو `QAction` يُكتب خارج `EnhancedTableView` هو **مخصص**.
- **يجب** أن يكون مصحوباً بتعليق `[CUSTOM]` يشرح السبب.
- أي "يدوي" بدون سبب واضح = **خطأ معماري** يجب تصحيحه.

**كيف تتعامل مع الكود اليدوي عند اكتشافه:**
1. ❓ اسأل: **لماذا** كُتب هكذا؟ هل هناك متطلب خاص؟
2. 🔍 افهم السياق قبل أي تعديل.
3. 🛠️ إذا كان يمكن إدخاله في الشجرة → أدخله.
4. 📝 إذا كان لا يمكن → وثقه كـ `[CUSTOM]`.
5. **⛔ لا تحذف ولا تُدمج قبل الفهم الكامل.**

## 🎨 القاعدة الخامسة — فلسفة الواجهة (UI Philosophy)
- **النصوص أولاً:** كل الأزرار تعتمد النص (Text Only) كقاعدة.
- **لا Emojis في الكود:** الرموز التعبيرية في واجهة المستخدم ممنوعة في مكونات النظام.
- **لغة البرنامج:** الفرنسية (French) هي اللغة الرسمية للبرنامج - الواجهة **والتعليقات**.
- **أسماء التبويبات:** مختصرة وواضحة (مثال: "Trésorerie" وليس "💰 Trésorerie Finance").
- **التوحيد البصري:** ما يُغيَّر في المكون الأساسي يُغيَّر في كل البرنامج.
- **التعليقات:** **ممنوعة** بالعربية - يجب استخدام الفرنسية فقط في كل الكود.
  - ✅ الصواب: `# [CUSTOM] Dialog personnalisé pour les transactions`
  - ❌ ممنوع: `# [CUSTOM] حوار مخصص للمعاملات`
  - ✅ الصواب: `# [WHY]: Nécessite un compte DA uniquement`
  - ❌ ممنوع: `# [WHY]: يحتاج حساب DA فقط`

## 💱 القاعدة الثامنة — العملات هي عمود البرنامج (Multi-Currency is Core)
**هذا البرنامج يعتمد بشكل جوهري على التعامل بالعملات المتعددة.**

### المبادئ الإلزامية:
1. **لا تعديل أو حذف عملة نشطة** إذا كانت مرتبطة بأي عملية مالية (شراء، دفع، دين).
2. **كل عملة نشطة** تُنشئ حساباً تلقائياً في الخزينة (`TreasuryAccount`). هذا سلوك **آلي** لا تُغيّره.
3. **أسماء حسابات العملات** قابلة للتخصيص، لكن ربطها بالعملة ثابت في قاعدة البيانات.
4. **تغيير سعر الصرف** لا يُعدِّل العمليات الماضية — العمليات التاريخية محمية بالكامل.
5. **أي منطق جديد يمس العملات** يجب عرضه على المالك أولاً لأن التأثير يتشعب عبر: الخزينة، العملاء، الشركاء، الديون الخارجية.

### خريطة تأثير العملة:
```
إضافة/تفعيل عملة جديدة
         ↓
 TreasuryAccount يُنشأ تلقائياً
         ↓
يظهر في: الخزينة + العملاء + الشركاء + الديون الخارجية
```

### ⛔ ممنوع منعاً باتاً:
- حذف عملة لها عمليات مالية مسجلة.
- تغيير `currency_code` لعملة موجودة (يكسر كل المراجع).
- إضافة حساب خزينة يدوياً لعملة بدون المرور عبر `toggle_world_currency()`.

## 💾 القاعدة التاسعة — النسخ الاحتياطي إلزامي (Backup Before Any Change)

### النظامان المتوفران:

**1. النسخ الاحتياطي التلقائي للبيانات (Automatic DB Backup)**
موجود ومفعَّل عبر `SettingsService`:
```python
# يعمل تلقائياً يومياً عند فتح البرنامج
service.perform_daily_backup()  # → backups/daily_backup_YYYY-MM-DD.db

# يُنفَّذ تلقائياً قبل أي Reset
service.create_backup(f"pre_reset_backup_{timestamp}.db")

# يُنفَّذ تلقائياً قبل أي Restore
service.create_backup(f"before_restore_{timestamp}.db")
```
النسخ تُحفظ في مجلد `backups/` (مستثنى من Git).

**2. Git — نظام إدارة الكود (Version Control)**
- يتتبع كل تغيير في الكود (ليس البيانات).
- يسمح بالرجوع لأي نقطة زمنية.
- **قاعدة Git الذهبية:** قبل أي تعديل معماري كبير →
```bash
git add .
git commit -m "BACKUP: قبل [وصف التعديل]"
```

### جدول المسؤوليات:
| ما يُحمى | الأداة | تلقائي؟ |
|----------|--------|---------|
| بيانات قاعدة البيانات | `backups/*.db` | ✅ نعم |
| كود البرنامج | Git | 🔵 يدوي (قبل كل تعديل كبير) |
| قرارات معمارية | `CHANGELOG.md` | 🔵 يدوي |

## ⚡ القاعدة العاشرة — قاعدة الإضافة السريعة (Smoothness: Quick Add)
**المبدأ:** لضمان سلاسة إدخال البيانات وعدم إجبار المستخدم على مغادرة النوافذ لإنشاء بيانات مرتبطة.
- **التنفيذ المبني:** في أي نافذة تحتوي على `dropdown` لجهة خارجية (عميل، مورد، جهة اتصال)، يجب توفير زر `[+]` لإنشاء الجهة مباشرة من النافذة باستخدام `SmartFormDialog`.
- **الاستخدام البرمجي:**
  - القوائم العادية: استخدام `from components.dialogs import create_quick_add_layout`.
  - داخل الـ `SmartFormDialog` (عبر Schema): إضافة مفتاح `quick_add_callback` إلى تعريف حقل الـ `dropdown`. يُمرر المحرك الـ `widget` (الـ ComboBox) إلى الـ callback الممرر تلقائياً لتحديث القائمة المنسدلة بدون إعادة تحميل النافذة الخلفية.

## 📋 القاعدة الحادية عشرة — إدارة الكتالوجات (Catalog Management Patterns)
**المبدأ:** أي نموذج يحتاج إلى جدول (CRUD) بسيط لإدارة قائمة خيارات (مثل البضائع، أنواع المصاريف، الخ) يجب ألا يُبنى من الصفر في كل مرة لتجنب التكرار البصري.
- **التنفيذ المبني:** استخدام مكوّن `GenericCatalogDialog` في `components/` والذي يستقبل:
  1. دوال (Callbacks) لجلب البيانات، إضافتها، حذفها، واستعادتها.
  2. عنوان النافذة (Title).
  3. أسماء الأعمدة (Headers).
- **الاستخدام البرمجي:** بدلاً من كتابة نافذة جديدة، نُعرّف الكتالوج الجديد في سطرين فقط بتمرير دوال الـ Service.

## 💳 القاعدة الثالثة عشرة — عرض الرصيد في المعاملات المالية (Balance Display in Transactions)
**المبدأ:** كل معاملة مالية (دفع/استلام) يجب أن تعرض رصيد العميل/المورد تلقائياً عند الاختيار، مع simulation للمبلغ المُدخَل.

**التنفيذ المبني:** استخدام `BaseTransactionDialog` - مكون قابل لإعادة الاستخدام:
```python
from components.base_transaction_dialog import BaseTransactionDialog

class MyTransactionDialog(BaseTransactionDialog):
    def __init__(self, parent=None):
        super().__init__(parent)
        self.set_entity_config(
            label="Client:",
            get_entities_func=lambda: [(c['name'], c['id']) for c in service.get_all_customers()],
            get_balance_func=lambda cid: service.get_customer_balance(cid),
            entity_name="Client",
            currency="DA"
        )
        self._setup_ui()
```

**المعاملات المطلوب تطبيقها:**
1. ✅ استلام الدفع من العملاء (ReceivePaymentDialog)
2. ⏳ إضافة بضائع عملاء
3. ⏳ دفع للموردين (موردين عملة)
4. ⏳ شراء العملة
5. ⏳ تسديد الديون الخارجية
6. ⏳ معاملات الشركاء

## 🎯 القاعدة الرابعة عشرة - الهدف النهائي (Profitability Focus)

**المبدأ:** البرنامج يُصمم ليجيب على سؤال واحد:
> "هل الشركة رابحة؟ كم؟ ولماذا؟"

**الهدف الاستراتيجي:** الوصول للربحية الحقيقية (Real Profitability)

**التنفيذ المبني:**

1. **تصنيف كل عملية مالية:**
   - إيراد (Revenu) ← Treasury, Customers, Partners
   - مصروف (Dépense) ← Logistics, Licenses, External Debt
   - استثمار (Investissement) ← Licenses, Containers

2. **Dashboard الربحية:**
   - ملخص الإيرادات والمصروفات
   - الرصيد الصافي (Net)
   - اتجاه الربحية (تصاعدية/تناقصية)

3. **تقرير الربحية الشامل:**
   - شهري + سنوي
   - تفصيل حسب القسم/العملة
   - مقارنة بالفترات السابقة

**الكود المرجعي:**
```python
# كل عملية مالية يجب أن تصنف
TRANSACTION_TYPE_REVENU = "REVENU"  # إيراد
TRANSACTION_TYPE_EXPENSE = "EXPENSE"  # مصروف
TRANSACTION_TYPE_INVEST = "INVEST"  # استثمار
```
